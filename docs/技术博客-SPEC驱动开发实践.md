## 0. 前言：从"唯技术论"到"文档先行"

### 0.1 一个资深工程师的 LLM-Based SDD 反思

2024 年，当 ChatGPT 引领的大模型浪潮席卷技术圈时，大多数人开始思考：**程序员是否会被取代？**

作为一个写了十年 Java 的老兵，我曾经坚信：设计文档不过是"形式主义"，真正的高手都是直接写代码。"ChatGPT 都能生成代码了，谁还花时间写文档？"

然而，经历了 OperatorBoard 这个 Multi-Agent 系统的从零构建后，我的观点发生了根本性转变——**LLM 确实能生成代码，但它无法替代系统性的设计思考。**

### 0.2 我们的教训

回顾 OperatorBoard 的开发历程，有太多"事后才明白"的坑：

| 时间节点 | 教训 | 反思 |
|---------|------|------|
| 第一周 | 直接写代码，边写边改 | 架构没有想清楚，通讯协议来回改了三版 |
| 第三周 | "这个接口设计不太对" | 缺乏 API 契约意识，前后端接口不匹配 |
| 第一个月 | 需求变更导致大量重构 | 没有需求追溯表，改一处漏三处 |
| 第三个月 | 代码注释过时，文档丢失 | "代码即文档"在现实中不成立 |
| 第六个月 | 新人上手困难 | 隐性知识没有显性化，知识传承断层 |

这些教训让我们深刻认识到：**没有文档的系统，终究是一盘散沙。**

## 1. 经典案例启示：为什么顶级 API 都重视 SPEC

### 1.1 Stripe：把 API 当产品来设计

Stripe 是 API 设计的标杆，其核心理念值得借鉴：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         Stripe API Design Philosophy                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 开发者体验优先 (Developer Experience First)                            │
│     ├── 一致性：所有 API 返回格式统一                                       │
│     ├── 可发现性：通过 IDE 插件和文档自动补全                              │
│     └── 可预测性：相同操作有相同模式                                       │
│                                                                             │
│  2. 错误即文档 (Errors as Documentation)                                  │
│     ├── 每条错误都有唯一码和详细说明                                       │
│     ├── 错误响应包含排查建议                                               │
│     └── 错误码体系完整且稳定                                               │
│                                                                             │
│  3. 版本即契约 (Versioning as Contract)                                    │
│     ├── 主版本号不兼容                                                     │
│     ├── 废弃有明确时间线                                                   │
│     └── 变更日志详尽                                                       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**关键洞察**：Stripe 的成功不是偶然的——他们把 API 当作产品来经营，每个细节都经过精心设计。

### 1.2 Twilio：用场景化组织代替技术化堆砌

Twilio 的 API 文档以**场景**为导向，而非以**技术组件**为导向：

| 传统文档组织 | Twilio 文档组织 |
|-------------|----------------|
| 按功能模块 | 按使用场景 |
| 技术视角 | 业务视角 |
| 开发者需要自己组合 | 给出完整解决方案 |

```
传统模式:
GET /sms/messages
POST /sms/messages
GET /calls
POST /calls
...

Twilio 模式:
├── 发送短信验证码
│   └── 代码示例 + API 调用 + 最佳实践
├── 呼叫中心集成
│   └── 完整呼叫流程 + 事件处理
└── IVR 语音菜单
    └── 状态机设计 + 录音处理
```

### 1.4 微软：分层文档体系

微软的文档体系以**分层**和**职责**著称：

| 层级 | 文档 | 关注点 |
|------|------|--------|

| 接口层 | API Spec | 契约定义 |
| 质量层 | Test Spec | 验证标准 |

## 2. LLM 时代的新挑战：Prompt 能否替代 SDD？

### 2.1 传统 SDD 与 LLM 生成代码的对比

| 维度 | 传统 SDD | LLM 生成代码 |
|------|----------|--------------|
| **覆盖范围** | 全局性系统设计 | 局部代码实现 |
| **抽象层次** | 高层设计，关注架构 | 低层实现，关注细节 |
| **决策追溯** | 记录了"为什么" | 只有"是什么" |
| **一致性** | 强一致性保障 | 每次生成可能不一致 |
| **沟通价值** | 团队共识载体 | 难以形成共识 |

**关键问题**：LLM 生成代码时没有"上下文感知"，它不知道你的系统架构、团队约定、未来演进方向。

### 2.2 LLM Prompt 的局限性

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         LLM Prompt 的局限性                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 缺乏系统性                                                              │
│     ├── Prompt 只能描述单个任务                                             │
│     ├── 无法表达系统各组件关系                                              │
│     └── 难以覆盖非功能性需求                                                │
│                                                                             │
│  2. 缺乏上下文连续性                                                        │
│     ├── 每次对话是独立的                                                    │
│     ├── 无法理解"项目背景"                                                 │
│     └── 历史决策丢失                                                        │
│                                                                             │
│  3. 缺乏设计模式的内化                                                      │
│     ├── 不理解"为什么要这样设计"                                            │
│     ├── 只会套用常见模式                                                    │
│     └── 无法权衡取舍                                                        │
│                                                                             │
│  4. 产出质量不稳定                                                          │
│     ├── 同一 Prompt 多次生成可能不同                                       │
│     ├── 边界情况可能遗漏                                                   │
│     └── 需要人工 review                                                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.3 正确的姿势：LLM + SDD

LLM 时代，SDD 不是被取代了，而是更重要了——因为 SDD 为 LLM 提供了**上下文**和**约束**。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                      LLM + SDD = 最佳实践                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│                              ┌─────────────────┐                           │
│                              │   SDD 文档       │                           │
│                              │  (上下文约束)    │                           │
│                              └────────┬────────┘                           │
│                                       │                                     │
│                                       ▼                                     │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                        LLM 生成代码                                  │   │
│  │                                                                       │   │
│  │   Prompt: "按照 SDD 中的架构设计，生成 XXX 模块代码"                 │   │
│  │               │                                                      │   │
│  │               ▼                                                      │   │
│  │   输出: 符合架构约束的、高质量的代码                                 │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  SDD 提供的价值:                                                            │
│  ├── 系统边界与组件关系                                                    │
│  ├── API 契约与数据模型                                                   │
│  ├── 质量标准与约束条件                                                    │
│  └── 决策历史与设计原理                                                    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.3 文档与代码的同步机制

| 实践 | 说明 |
|------|------|
| **Docs near Code** | 文档放在代码库中，与代码同一 repo |
| **版本对应** | 文档版本与代码版本对应 (如 SDD v1.3 对应 release-1.3) |
| **变更联动** | 代码 PR 必须更新对应文档 |
| **自动化检查** | CI 中检查文档格式与完整性 |

### 4.2 关键架构决策 (ADRs)

我们在开发过程中形成了 5 个核心 ADR：

| ADR | 决策 | 驱动因素 |
|-----|------|----------|
| ADR-001 | 采用 FastAPI 作为 Agent API 框架 | 异步支持 + Pydantic 验证 + 自动 OpenAPI |
| ADR-002 | Java 层采用 CQRS 模式 | 读写分离 + NL2SQL 命令与查询分离 |
| ADR-003 | 采用 MiniMax M2-her LLM 意图识别 | 复杂查询理解 + 中文处理 |
| ADR-004 | NL2SQL 使用自托管 SQLCoder | 数据安全 + 成本控制 |
| ADR-005 | 采用 Block 格式封装响应 | 流式输出 + 多类型内容混合 |

## 5. 过程中的问题与挑战

### 5.1 技术挑战

#### 5.1.1 Multi-Agent 协作模式设计

**问题描述**:
OperatorBoard 是一个 Multi-Agent 系统，最初我们没有想清楚 Agent 之间的协作模式。

**探索过程**:

```
阶段1: 链式模式
Agent1 → Agent2 → Agent3 → 用户
问题: Agent2 失败导致整个链条中断

阶段2: 星型模式
        → Agent2 →
       ↗        ↘
用户 ←         Agent4 → 外部服务
       ↘        ↙
        → Agent3 →
问题: 中心节点过载，扩展性差

阶段3: 流水线模式 (最终选择)
Planner → Specialist → Verifier
问题: 需要清晰的职责边界

最终采用 Skill 化的编排模式 (参考 Skill 7: Multi-Agent 协作三问)
```

**解决方案**:
引入 Skill 7 的 Multi-Agent 协作框架，明确三个核心问题：
1. **结果能否复用？** - Specialist Agent 产出可供其他 Agent 使用
2. **谁来负责路由？** - Planner Agent 负责分发，Verifier Agent 负责校验
3. **某个 Agent 失败怎么办？** - 降级策略 + 重试机制

**代码示例**:

```python
# operator-agent/router.py - 基于 SPEC 的路由实现
class Router:
    """
    路由器 - 依据 SDD.md 接口定义

    职责:
    - 解析 IntentDetection 结果
    - 路由到对应处理模块
    - 处理降级逻辑

    参考: SDD.md 4.2.2 NL2SQL查询流程
    """

    async def route(self, intent_result: IntentResult) -> Response:
        # 依据 SPEC 的路由表
        ROUTE_MAP = {
            Intent.SITE_DATA: self._handle_site_data,
            Intent.INDICATOR_DATA: self._handle_indicator_data,
            Intent.OPERATOR_LIST: self._handle_operator_list,
            Intent.NL2SQL: self._handle_nl2sql,
        }

        handler = ROUTE_MAP.get(intent_result.intent)
        if not handler:
            raise AgentAPIError(INTENT_NOT_SUPPORTED)
```

#### 5.1.2 NL2SQL 准确率问题

**问题描述**:
SQLCoder 生成的 SQL 有时会出现语法错误或语义偏差。

**解决过程**:

| 方案 | 尝试 | 效果 |
|------|------|------|
| 提示词优化 | 增加 Schema 描述 | 准确率 +15% |
| 结果校验 | 执行 SQL 后校验结果 | 语法错误减少 80% |
| 降级策略 | SQL 不可用时使用预设查询 | 用户体验保障 |

**ADR-004 决策**:

```markdown
# ADR-004: NL2SQL 使用自托管 SQLCoder

## 决策
- 采用自托管 SQLCoder 而非商业 API

## 后果
### 正面
- 数据安全有保障
- 成本可控

### 负面
- 准确率不如 GPT-4
- 需要额外校验层
```

### 5.2 管理挑战

#### 5.2.1 文档与代码同步问题

**问题描述**:
开发人员不愿意写文档，更新文档的频率远低于代码变更。

**解决过程**:

```
问题演进:
├── v1.0: "代码即文档" → 代码注释过时，文档丢失
├── v1.2: 开始补充文档 → 文档存在但与代码不一致
├── v1.4: 要求代码 PR 必须更新文档 → 执行度不高
└── v1.6: 引入自动化检查 + 文档覆盖率指标

解决措施:
1. Doc-as-Code: 文档放在代码仓库，版本对应
2. PR 联动: 代码变更必须更新对应 SPEC
3. 自动化检查: CI 中检查 SPEC 完整性
4. 指标驱动: 追踪文档覆盖率
```

**具体实践**:

```yaml
# .github/workflows/spec-check.yml

    # 检查 STD 测试用例覆盖率
```

### 5.3 流程挑战

#### 5.3.1 SPEC 评审流于形式

**问题描述**:
初期 SPEC 评审变成了"走过场"，评审意见不被采纳。

**解决过程**:

```
问题识别:
├── 评审时间短 (15分钟)
├── 评审意见无跟踪
├── 评审后随意变更 SPEC
└── 评审者 vs 实现者分离

解决措施:
1. 评审 checklist 强制检查
2. 评审意见必须在 SPEC 中有记录
3. 评审后变更需要再次评审
4. 评审者参与实现过程
```

**评审 checklist 示例**:

```markdown
## SPEC 评审 Checklist

□ 1. 完整性检查
   □ 所有需求都有对应设计
   □ API 端点都有输入/输出定义
   □ 错误码都有定义

□ 2. 一致性检查
   □ 模块间接口一致
   □ 数据模型无歧义
   □ 命名规范统一

□ 3. 可实现性检查
   □ 技术方案可行
   □ 性能目标可达
   □ 依赖组件明确

□ 4. 可测试性检查
   □ 每个功能点都有测试用例
   □ 验收标准明确可测
   □ 边界条件已覆盖
```

#### 5.3.2 变更管理困难

**问题描述**:
开发过程中需求变更频繁，SPEC 跟不上变化。

**解决方案**:

```
变更分类处理:
├── P0 紧急变更: 先改代码后补 SPEC (但必须当天补)
├── P1 重要变更: 评审后实施
└── P2 普通变更: 纳入迭代计划

变更流程:
1. 提出变更申请
2. 评估影响范围
3. 确定变更分类
4. 执行对应流程
5. 更新 SPEC 版本
6. 通知相关方
```

```
│   │    │              │                                                 │  │
│   │    │              ▼                                                 │  │
│   │    │          自动检测不一致 → 更新 SPEC                            │  │
│   │    │                                                                 │  │
│   │    ▼                                                                 │  │
│   │  测试验证 ←────────────────────────────────────────────────┘         │  │
│   │                                                                      │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│   最终目标:                                                               │
│   ├── 代码变更自动检测 → 更新 SPEC                                       │
│   ├── SPEC 变更自动验证 → 测试通过                                        │
│   └── 文档即代码，代码即文档                                              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 5.2 API 设计规范

遵循 Stripe 的 API 设计原则：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        API Design Checklist                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  □ 1. 资源命名                                                              │
│     ├── 使用复数名词 (POST /operators, GET /operators/{id})              │
│     ├── 小写下划线分隔 (operator_name, not operatorName)                   │
│     └── 层级清晰 (/operators/{id}/indicators)                              │
│                                                                             │
│  □ 2. HTTP 方法语义                                                         │
│     ├── GET: 幂等读取                                                       │
│     ├── POST: 创建资源                                                      │
│     ├── PUT: 全量更新                                                       │
│     ├── PATCH: 部分更新                                                     │
│     └── DELETE: 删除资源                                                    │
│                                                                             │
│  □ 3. 状态码规范                                                            │
│     ├── 200 OK                                                              │
│     ├── 201 Created                                                         │
│     ├── 400 Bad Request                                                     │
│     ├── 401 Unauthorized                                                    │
│     ├── 403 Forbidden                                                       │
│     ├── 404 Not Found                                                       │
│     └── 500 Internal Server Error                                           │
│                                                                             │
│  □ 4. 错误响应                                                              │
│     ├── 错误码 (error.code)                                                │
│     ├── 英文消息 (error.message_en)                                        │
│     ├── 中文消息 (error.message_zh)                                        │
│     └── 详细信息 (error.detail)                                             │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 5.3 ADR 编写规范

每个 ADR 包含：

| 章节 | 内容 |
|------|------|
| **状态** | Proposed / Accepted / Deprecated / Superseded |
| **上下文** | 决策背景和问题陈述 |
| **决策** | 具体的决策内容 |
| **后果** | 正面 / 负面 / 中性后果 |
| **替代方案** | 考虑过但未选择的方案及原因 |

## 9. 总结：文档驱动开发的黄金法则

### 6.1 核心理念

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    文档驱动开发的七条黄金法则                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 文档刚够好 (Just Barely Good Enough)                                  │
│     不要过度文档化，只写"刚刚好"需要的文档。                                │
│                                                                             │
│  2. 文档跟着代码走 (Docs near Code)                                         │
│     文档放在代码仓库中，确保版本对应、同步更新。                            │
│                                                                             │
│  3. 每个文档单一职责 (Single Responsibility)                                │
│     PRD 回答 Why 和 What，SDD 回答 How，STD 回答 Verify。                  │
│                                                                             │
│  4. API 即契约 (API as Contract)                                            │
│     API 文档是前后端的唯一真相来源，变更需要评审。                          │
│                                                                             │
│  5. 决策需要记录 (Decisions Need Documentation)                            │
│     ADR 记录架构决策及原因，便于追溯。                                      │
│                                                                             │
│  6. 测试驱动文档 (Test Drives Documentation)                               │
│     测试用例验证文档准确性，文档描述需要可测试。                            │
│                                                                             │
│  7. 文档是团队知识资产 (Documentation as Knowledge Asset)                    │
│     文档是团队共同的知识沉淀，不是个人财产。                                │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```
