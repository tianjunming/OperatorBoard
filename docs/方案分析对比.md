# 软件开发文档体系方案分析对比

**文档版本**: 1.0
**编制日期**: 2026-04-12
**参考标准**: IEEE 829 | Agile Manifesto | Microsoft Docs | ThoughtWorks

---

## 1. 概述

本文档对比业界主流软件开发文档体系，为 OperatorBoard 项目文档拆分提供参考依据。

---

## 2. 方案对比

### 2.1 主流方案总览

| 方案 | 文档数量 | 核心文档 | 适用场景 | 代表企业 |
|------|---------|---------|---------|---------|
| **IEEE 829** | 5-8个 | SRS, SSDS, STD, SAD | 传统瀑布、CMMI认证 | 电信、金融、军工 |
| **敏捷轻量** | 2-3个 | PRD, Working Software | 互联网、Startup | Spotify, Buffer |
| **微软式** | 4-6个 | PRD, Functional Spec, Design | 企业级、混合开发 | Microsoft |
| **Google式** | 1个 | Design Doc | 工程师驱动文化 | Google |
| **阿里式** | 6-8个 | PRD, AD, API, TC, Runbook | 大型团队、电商 | Alibaba |

## 3. 详细方案分析

### 3.1 IEEE 829 标准体系

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    IEEE 829 Software Documentation Tree                  │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  SDP (Software Development Plan)                                       │
│  └─ 项目管理计划: 里程碑、资源、风险管理                                  │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                        SRS (Software Requirements)                │   │
│  │  1. Introduction          - 目的、范围、定义                      │   │
│  │  2. Overall Description  - 产品概述、用户特征、约束               │   │
│  │  3. Specific Requirements - 功能需求、接口需求、性能需求          │   │
│  │  4. Appendices           - 索引、术语表                           │   │
│  │  5. Index               - 需求索引                               │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                    SSDS (Software Design Description)             │   │
│  │  1. Overview           - 设计概述、模块划分                       │   │
│  │  2. Subsystem Design  - 各子系统详细设计                         │   │
│  │  3. Interface Design  - 外部/内部接口                            │   │
│  │  4. Data Design       - 数据库/文件设计                          │   │
│  │  5. appendices        - 支持信息                                 │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                    STD (Software Test Documentation)              │   │
│  │  1. Test Plan        - 测试策略、资源、进度                       │   │
│  │  2. Test Case Spec   - 测试用例详细规格                          │   │
│  │  3. Test Proc Spec   - 测试规程                                 │   │
│  │  4. Test Item Trans  - 测试项可追溯性                           │   │
│  │  5. Test Log         - 测试日志                                  │   │
│  │  6. Test Summary     - 测试总结                                  │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  SAD (Software Assurance Document)                                      │
│  └─ 质量保证: 验证计划、确认方法、评审记录                               │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**优点**:
- 标准化程度高，业界广泛认可
- 覆盖完整，从需求到测试全链路
- 易于CMMI等体系认证

**缺点**:
- 文档量大，敏捷场景过于笨重
- 更新维护成本高
- 与代码分离，难以保持同步

**适用场景**: 传统行业、军工、金融、需要认证的组织

### 3.2 敏捷轻量方案

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    Agile "Just Barely Good Enough"                       │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   "Working software over comprehensive documentation"                    │
│   "The best documentation is working code"                              │
│                                                                         │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │                    Investment Triangle                          │   │
│   │                                                                   │   │
│   │                        Executable                                 │   │
│   │                         Code                                      │   │
│   │                          ▲                                        │   │
│   │                         ╱ ╲                                       │   │
│   │                        ╱   ╲                                      │   │
│   │                       ╱     ╲                                     │   │
│   │                      ╱       ╲                                    │   │
│   │                     ╱_________╲                                   │   │
│   │              Tests  ◄   Living   ►  Documentation                │   │
│   │                     Documentation                                 │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│   文档优先级:                                                            │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │  1. User Stories + Acceptance Criteria (最重要的文档)           │   │
│   │  2. README & Quick Start (新人上手)                             │   │
│   │  3. API Documentation (自动生成)                               │   │
│   │  4. Architecture Decision Records (ADR)                       │   │
│   │  5. Vision/Roadmap (可选)                                       │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**优点**:
- 文档量少，轻量灵活
- 与代码保持同步
- 适应需求变化

**缺点**:
- 知识传承依赖人
- 新成员上手困难
- 不适合复杂系统

**适用场景**: 互联网产品、早期Startup、节奏快的团队

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    Microsoft Documentation Stack                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  Executive Summary (产品愿景)                                            │
│  └─ 一页纸: 市场定位、核心价值、关键指标                                  │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                    Requirements & Specifications                  │   │
│  │                                                                   │   │
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐               │   │
│  │  │   PRD       │  │  Functional │  │  Non-Func   │               │   │
│  │  │  (What)    │  │  Spec       │  │  Spec       │               │   │
│  │  │  用户视角   │  │  技术视角   │  │  性能约束   │               │   │
│  │  │            │  │             │  │             │               │   │
│  │  │  解决什么问题│  │  如何实现   │  │  做到什么程度│               │   │
│  │  └─────────────┘  └─────────────┘  └─────────────┘               │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                       Design Documents                            │   │
│  │                                                                   │   │
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐               │   │
│  │  │  Architecture│  │  API Spec   │  │  UI/UX     │               │   │
│  │  │  Document   │  │  (OpenAPI)  │  │  Design    │               │   │
│  │  │            │  │             │  │            │               │   │
│  │  │  系统架构   │  │  接口定义   │  │  界面设计   │               │   │
│  │  └─────────────┘  └─────────────┘  └─────────────┘               │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**优点**:
- 分层清晰，职责明确
- 兼顾用户视角和技术视角
- 适合中大型团队

**缺点**:
- 文档数量仍然较多
- 需要专人维护

**适用场景**: 企业级应用、混合开发模式

### 3.4 Google 方案

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    Google Engineering Practices                           │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   "Design Docs are the primary vehicle for design review"              │
│                                                                         │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │                    Design Doc Template                            │   │
│   │                                                                   │   │
│   │  1. Title & Status                                                │   │
│   │     - 标题、作者、日期、状态 (Draft/In Review/Approved)           │   │
│   │                                                                   │   │
│   │  2. Summary                                                       │   │
│   │     - 一句话总结                                                    │   │
│   │                                                                   │   │
│   │  3. Background                                                    │   │
│   │     - 背景、动机、解决的问题                                        │   │
│   │                                                                   │   │
│   │  4. Design                                                        │   │
│   │     - 详细设计方案                                                 │   │
│   │     - 包含: 系统图、API设计、数据模型、流程图                      │   │
│   │                                                                   │   │
│   │  5. Alternatives Considered                                       │   │
│   │     - 替代方案及否决原因                                           │   │
│   │                                                                   │   │
│   │  6. Open Questions                                               │   │
│   │     - 未解决的决策                                                 │   │
│   │                                                                   │   │
│   │  7. Conclusion                                                   │   │
│   │     - 结论和下一步                                                │   │
│   │                                                                   │   │
│   │  8. Referenced Projects                                          │   │
│   │     - 相关项目和文档                                               │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│   特点:                                                                 │
│   - 一个文档包含所有设计信息                                              │
│   - 设计评审是必经流程                                                   │
│   - 文档保存在代码库 (而不是Wiki)                                        │
│   - 每次大的变更都需要新版本                                              │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**优点**:
- 单一文档，维护简单
- 工程师驱动文化
- 设计评审流程保证质量

**缺点**:
- 对工程师素质要求高
- 可能遗漏细节

**适用场景**: 工程师驱动型团队、技术实力强的组织

```
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**优点**:
- 分工明确，适合大团队
- 流程规范
- 知识传承好

**缺点**:
- 文档量大
- 流程重
- 小团队难以承受

**适用场景**: 大型团队、电商、金融

## 4. 方案对比矩阵

### 4.1 维度对比

| 维度 | IEEE 829 | 敏捷轻量 | 微软式 | Google式 | 阿里式 |
|------|---------|---------|--------|---------|--------|
| **文档数量** | 5-8 | 2-3 | 4-6 | 1 | 6-8 |
| **维护成本** | 高 | 低 | 中 | 中 | 高 |
| **适用团队规模** | 大 | 小 | 中大 | 中 | 大 |
| **学习曲线** | 陡 | 平缓 | 中 | 平缓 | 陡 |
| **技术深度** | 高 | 低 | 高 | 高 | 高 |
| **业务覆盖** | 全 | 轻 | 全 | 中 | 全 |
| **认证友好** | 是 | 否 | 部分 | 否 | 部分 |
| **灵活性** | 低 | 高 | 中 | 中 | 低 |

## 5. 我们的选择

### 5.1 OperatorBoard 文档体系

基于对各方案的分析，我们采用 **IEEE 829 + 敏捷轻量** 混合模式：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    OperatorBoard 文档体系                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  SPEC.md (文档索引)                                                      │
│  └─ 文档导航入口，关联各分文档                                           │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                        需求文档层                                  │   │
│  │                                                                   │   │
│  │  ┌─────────────┐                                                │   │
│  │  │   PRD.md    │  Product Requirements Document                │   │
│  │  │  需求分析   │  功能需求、非功能需求、验收标准                   │   │
│  │  └─────────────┘                                                │   │
│  │                                                                   │   │
│  │  回答: Why (为什么做) & What (做什么)                             │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                        设计文档层                                  │   │
│  │                                                                   │   │
│  │  ┌─────────────┐  ┌─────────────┐                              │   │
│  │  │   SDD.md    │  │   CDD.md    │                               │   │
│  │  │  软件设计    │  │  代码设计   │                               │   │
│  │  │  架构/接口   │  │  实现细节   │                               │   │
│  │  └─────────────┘  └─────────────┘                              │   │
│  │                                                                   │   │
│  │  回答: How (如何实现)                                            │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                        测试文档层                                  │   │
│  │                                                                   │   │
│  │  ┌─────────────┐                                                │   │
│  │  │   STD.md    │  Software Test Document                        │   │
│  │  │  测试设计    │  测试策略、用例、验证标准                        │   │
│  │  └─────────────┘                                                │   │
│  │                                                                   │   │
│  │  回答: How to Verify (如何验证)                                   │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 5.2 选择理由

| 考量因素 | 我们的情况 | 选择依据 |
|---------|-----------|---------|
| **团队规模** | 中小型 (5-20人) | 微软式/敏捷轻量区间 |
| **项目复杂度** | 中等 (多Agent+NL2SQL) | 需要分层设计文档 |
| **质量要求** | 高 (电信级) | 需要测试文档 |
| **迭代节奏** | 敏捷 | 轻量但完整 |
| **知识传承** | 需要 | 文档体系支撑 |

### 5.3 文档职责划分

| 文档 | 主要读者 | 更新频率 | 维护责任人 |
|------|---------|---------|-----------|
| SPEC.md | 所有角色 | 重大变更 | Tech Lead |
| PRD.md | PM, 产品, Dev | 需求变更 | 产品经理 |
| SDD.md | 架构师, Dev | 设计变更 | 架构师 |
| CDD.md | Dev | 代码变更 | 开发人员 |
| STD.md | QA, Dev | 测试周期 | 测试工程师 |
